## 1. 风险评估要求什么——对照分析

将《网络数据安全风险评估办法》的典型评估流程拆解为7个步骤，逐项对照：

```
评估流程：
① 数据资产盘点 → ② 威胁识别 → ③ 脆弱性识别 
→ ④ 风险场景构建 → ⑤ 风险计算（可能性×影响） 
→ ⑥ 风险评价与排序 → ⑦ 整改跟踪
```

| 步骤 | 要求 | 我们现状 | 差距 |
|------|------|----------|------|
| **① 数据资产盘点** | 识别所有数据资产、数据量、存储位置、流转路径 | ✅ 已完成。fields + directories + datasources 覆盖了数据字段级别的盘点 | **无。这是我们的强项。** |
| **② 威胁识别** | 识别可能危害数据安全的外部/内部威胁源 | ❌ 不存在。没有威胁目录（Threat Catalog） | **需要从零构建。** 但可以复用国标——GB/T 35274-2023 和《网络数据安全风险评估指南》提供了标准化威胁清单 |
| **③ 脆弱性识别** | 识别数据在采集、存储、使用、共享、销毁各环节的技术和管理脆弱性 | ❌ 不存在。没有脆弱性检查项 | **中等差距。** anomaly_detector 只检测了"未映射"和"缺失信息"两种脆弱性，实际脆弱性远不止这些（如：未加密存储、过度权限、缺少脱敏等） |
| **④ 风险场景构建** | 将数据资产×威胁×脆弱性组合为具体风险场景 | ❌ 不存在。没有风险场景模型 | **核心差距。** 这是从"分类分级"到"风险评估"的关键一跳——需要建立"什么样的数据、面临什么威胁、因为什么脆弱性、可能导致什么后果"的推理链 |
| **⑤ 风险计算** | 评估风险的可能性和影响，计算风险值 | ❌ 不存在。没有风险量化模型 | **中等差距。** 合规引擎的"影响对象×危害程度"矩阵可以复用为"影响"维度的输入，但"可能性"维度需要全新构建 |
| **⑥ 风险评价与排序** | 按风险值排序，确定优先级 | ❌ 不存在 | **小差距。** 有排序逻辑就能做，本质是数据展示 |
| **⑦ 整改跟踪** | 制定整改措施、跟踪闭环 | ❌ 不存在。没有整改工作流 | **中等差距。** ReviewRecord 的审核工作流可以扩展为整改工作流，但需要新增整改任务模型 |

**总结：7步中我们完成了1步（资产盘点），2步有可以复用的基础（风险计算、整改跟踪），4步需要从零构建。**

## 2. 差距的根源——"分类分级"不等于"风险评估"

当前平台的定位是**数据分类分级**（回答：数据敏感程度如何？），风险评估要求的是**风险分析**（回答：数据可能出什么事？可能性多大？后果多严重？）。

```
分类分级：字段A → 个人权益 → 一般危害 → 敏感数据 ✅ 已完成
风险评估：字段A → 敏感数据 → 未加密存储 + 内部人员可访问 
         → 可能被内部人员泄露 → 可能性中 × 影响中 = 中等风险 ❌ 未完成
```

分类分级的结果是风险评估的**输入**（告诉我们哪些数据是重要的），但风险评估还需要两个额外的输入：威胁（谁会来攻击）和脆弱性（哪里容易出问题）。

## 3. 实现路径——分三期，不改现有代码，只做加法

### Phase 1：威胁与脆弱性知识库（1-2天）

**不写一行业务逻辑，只建数据模型 + 录入标准化的知识。**

新增三个模型：

```python
# ① 威胁目录 — 来自国标 GB/T 35274-2023
class Threat(Base):
    id, code, name, category,          # 威胁编码、名称、类别（外部攻击/内部泄露/供应链/物理）
    description, typical_scenario       # 描述、典型场景
    # 预置 ~30 条标准威胁（非法访问、数据泄露、内部人员恶意操作、第三方接口滥用...）

# ② 脆弱性检查项 — 来自等保2.0 + 数据安全能力成熟度模型
class VulnerabilityCheck(Base):
    id, code, category,                # 检查项编码、类别（采集/存储/使用/共享/销毁）
    check_item, description,           # 检查项名称、描述
    severity,                          # 严重程度（高/中/低）
    auto_detect_rule                   # 可自动检测的规则（JSON，可选）
    # 预置 ~40 条检查项（未加密存储、过度权限、缺少脱敏、未签数据协议...）

# ③ 风险场景库 — 威胁 × 脆弱性 × 数据类别 的组合
class RiskScenario(Base):
    id, threat_id, vuln_check_id,
    applicable_data_levels,            # 适用的数据级别（core/important/sensitive/normal均可）
    default_likelihood,                # 默认可能性（可被自动检测结果修正）
    description, mitigation_advice     # 场景描述、整改建议
    # 预置 ~50 个标准场景
```

**为什么先建知识库而不是先写引擎？** 因为威胁和脆弱性是高度标准化的——国标已经定义好了。先把它录进去，引擎只是"匹配"的问题。而且知识库本身就是可交付的——即使没有自动化，客户也可以对着检查项人工评估。

**Phase 1 产出：** 风险评估标准页面（只读），展示威胁目录+脆弱性检查项+风险场景库。人工评估时可以对照使用。

### Phase 2：自动化风险扫描引擎（2-3天）

**这是核心——把"数据资产×威胁×脆弱性→风险"的推理链自动化。**

```
RiskAssessmentEngine 的推理流程：

输入：一个 Field（已有分类分级结果）
  ↓
Step 1: 脆弱性自动检测
  - 遍历 VulnerabilityCheck，用 auto_detect_rule 判断该字段是否存在该脆弱性
  - 例：finance_data_level IN (core, important) AND NOT has_encryption_flag → 触发"未加密存储"脆弱性
  - 对于无法自动检测的检查项，标记为"需人工确认"
  ↓
Step 2: 威胁-脆弱性匹配
  - 对于被触发的每个脆弱性，从 RiskScenario 库中匹配适用的威胁
  ↓
Step 3: 风险计算
  - 可能性 = 威胁默认可能性 × 脆弱性严重程度系数 × 已有安全措施折扣
  - 影响 = 数据级别映射（core→极高、important→高、sensitive→中、normal→低）
  - 风险值 = 可能性 × 影响 → 四级：极高/高/中/低
  ↓
Step 4: 去重聚合
  - 如果同一字段在多个风险场景中都被识别，取最高风险等级
  ↓
输出：RiskAssessmentResult（字段级 + 聚合到表级/库级）
```

**可自动检测的脆弱性示例（~15条，其余需人工确认）：**

| 自动检测规则 | 检查逻辑 |
|-------------|----------|
| 核心/重要数据未加密 | `finance_data_level IN (core,important)` AND 字段未标注加密 |
| 敏感数据无脱敏策略 | `finance_data_level IN (sensitive)` AND 字段未标注脱敏 |
| 个人数据未分类 | 含PII关键词 AND `finance_category_id IS NULL` |
| 超额权限 | 字段标注"全员可访问" AND `finance_data_level IN (core,important)` |
| 过期数据未清理 | `status='active'` AND `created_at < 3年前` |
| 数据未映射到业务目录 | `finance_category_id IS NULL`（复用 anomaly_detector） |

**Phase 2 产出：** 
- `POST /api/risk/scan` — 运行风险评估扫描
- `GET /api/risk/results?field_id=&level=` — 查看评估结果
- `GET /api/risk/summary` — 风险总览（按等级分布、按脆弱性分布、按数据级别分布）

### Phase 3：评估报告 + 整改闭环（1-2天）

```python
# ④ 评估报告 — 自动生成Word/PDF
class AssessmentReport(Base):
    id, title, assessment_date, period_start, period_end,
    scope_description,    # 评估范围
    summary,              # AI生成的评估摘要
    total_risks, high_risks, medium_risks, low_risks,
    status,               # draft/submitted/acknowledged
    created_by, created_at

# ⑤ 整改任务 — 从ReviewRecord扩展
class RemediationTask(Base):
    id, risk_result_id,            # 关联的风险
    action, owner, due_date,       # 整改措施、责任人、期限
    status,                        # open/in_progress/completed/verified
    verification_evidence          # 验证证据
```

**Phase 3 产出：**
- 一键生成年度评估报告（复用现有 excel_service 的报告能力 + AI 生成评估摘要）
- 整改任务管理：创建→分配→执行→验证闭环
- 年度重评估提醒（基于 ImportantDataSnapshot 的时间戳，满一年自动提醒）

## 6. 产品化角度——"先卖什么"

不建议等三期全做完再去触达客户。每一期都是可独立交付的：

- **Phase 1 完成后 → 数据安全风险评估知识库。** 客户可以对着标准检查项做人工评估。这本身就是一个产品——"帮你省掉翻国标的时间"。定价：免费，作为引流工具。
- **Phase 2 完成后 → 自动化风险评估工具。** 15项自动检测 + 人工确认其余项。核心卖点：从"每年花几周做评估"变成"一键扫描+人工复核"。定价：模块附加费。
- **Phase 3 完成后 → 端到端评估报告+整改平台。** 从扫描→报告→整改→次年重评估，全流程覆盖。核心卖点：满足《办法》的年度评估合规要求，审计留痕。定价：年度订阅。

## 附录A：产品定位讨论

### 监管现实

金融机构的数据安全风险评估报告，监管要求的是**独立第三方出具**——会计师事务所或专业安全评估机构签字盖章。我们平台生成的东西，监管不认。

### 我们在哪

```
一年周期中：
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ 持续监控  │  │ 评估准备  │  │ 外部评估  │  │ 整改跟踪  │
│ (11个月)  │  │ (1个月)   │  │ (3-4周)   │  │ (持续)    │
└──────────┘  └──────────┘  └──────────┘  └──────────┘
     ↑             ↑            ↑              ↑
  ✅ 我们的平台  ✅ 我们的平台  ❌ 事务所出报告  ✅ 我们的平台
    自动化分类     一键生成       监管只认        整改任务
    脆弱性检测     预评估底稿     第三方签章      闭环追踪
    实时预警       差距分析
```

**我们覆盖 12 个月里的 11.5 个月。那 0.5 个月（第三方签字），我们不碰。**

### 三方价值

**对金融机构：** 评估准备从 2-3 人×4-6 周缩至 1 周。咨询费减少 30-50%。整改从"Excel 跟踪"变成"平台闭环"。

**对事务所（渠道，不是对手）：** 省掉初级顾问的数据盘点工时（利润最低的环节）。同一个项目，利润率提升或降价抢单。可白标授权给事务所，在多个客户间复用。

**对监管：** 隐含诉求不是"每年交一份报告"，而是"机构真正在管数据安全"。一年的持续监控记录比一份年度报告更能证明。

### 一句话定位

> **让金融机构在外部评估前就发现问题、修好问题。让事务所的评估从"从零发现"变成"验证已发现"。**

类比：体检前自己先量血压、测血脂——不是替代医生诊断，而是让医生时间花在判断上而不是测量上。

### 市场格局

2025-2026 年，数据安全平台已从"工具型"演进为"合规驱动的泛监测体系"。头部厂商在金融行业实现 >95% 敏感数据识别准确率、>90% 人工替代率。

### 第一梯队（综合型）

| 厂商 | 核心能力 | 金融客户证据 | 对标我们的差异 |
|------|----------|-------------|---------------|
| **奇安信** | 零信任+数据流动监测+UEBA | 国有大行敏感操作拦截率99.3% | 重"防护"，我们重"评估准备"——不是竞争是互补 |
| **全知科技** | 以API为核心的数据安全，自研多模态AI | 中国人寿财险API风险实时监测 | 重"API和流动"，我们重"数据本身的分级+评估" |
| **启明星辰** | 合规驱动的联动治理，合规模板全 | 政务/运营商成熟，金融在拓展 | 合规审计证据链强，但偏"事后审计" |

### 第二梯队（云原生/专项型）

| 厂商 | 核心能力 | 定价参考 |
|------|----------|----------|
| **腾讯云DSGC** | AI+规则双引擎分类分级，金融模板，免费试用6个月 | TCO预估降70% vs自建 |

| **绿盟IDR** | 敏感数据发现+风险评估，首次入围Forrester | — |
| **安恒信息** | "恒脑"安全大模型深度AI注入 | — |

### 空白地带：我们的可能位置

| 维度 | 现有竞品 | 我们 |
|------|---------|------|
| **定位** | 数据安全防护/监测/审计 | 数据安全**评估准备** + 持续监控 |
| **与事务所关系** | 竞争（替代审计）或无关 | **渠道合作**（白标给事务所） |
| **核心功能** | 分类分级 → 风险告警 → 处置 | 分类分级 → 评估准备 → **预评估底稿导出** → 整改闭环 |
| **AI角色** | 规则引擎为主，AI辅助 | AI自动生成评估摘要+差距分析 |
| **目标客户** | 大中金融机构安全部门 | 中小金融机构合规部门（人手更少、需求更痛）|
| **定价区间** | 50-200万/年（头部）| 5-20万/年（附加模块或常年服务） |

**差异化假设（待验证）：**
1. 竞品都在做"安全防护"——实时监测、告警、阻断。我们在做"评估准备"——这是一个不同的采购场景（合规部预算 vs 安全部预算）
2. 竞品的目标是替代或减少人在安全运营中的参与。我们的目标是替代评估准备中的人工盘点
3. 中小金融机构养不起一个数据安全团队——但监管要求他们每年做评估。他们的痛点是"没人做、不会做"——不是"攻防不够强"
4. 与事务所的渠道合作是竞品没做的——头部安全厂商和事务所是竞争关系，我们是合作关系
