## 二、分类分级标准详解

### 2.1 三层分类体系

```
一级分类（3类）
├── 业务数据
│   ├── 金融市场数据 ─── 股票、债券、基金、外汇、商品、期货期权等（52类三级）
│   ├── 宏观经济数据
│   ├── 组织机构数据
│   ├── 行业指标数据
│   └── 资讯报告数据
├── 用户数据
│   ├── 个人用户数据 ─── 基本信息、交易数据、生物特征识别信息（3类）
│   └── 机构用户数据 ─── 基本信息、交易数据（2类）
└── 企业数据
    ├── 经营管理数据 ─── 财务、结算、人力、市场、风控等（6类）
    └── 系统运维数据 ─── 配置、日志、安全监测、安全事件（4类）
```

**三级合计：52 + 5 + 10 = 67 类。** 附录A为每类提供描述、示例和参考最低级别。

### 2.2 四级分级标准

| 级别 | 名称 | 判定标准 | 平台映射 |
|------|------|---------|---------|
| **第1级** | 核心数据 | 对国家安全、经济运行造成**特别严重**危害 | 相当于 L4+（超出现有 L4 范围） |
| **第2级** | 重要数据 | 造成**严重**危害 | ≈ L4（机密/严格保密） |
| **第3级** | 敏感一般数据 | 对经济运行/社会稳定/公共利益有**一定影响**，或对组织/个人造成**重要影响** | ≈ L3（敏感） |
| **第4级** | 常规一般数据 | 对组织权益、个人权益造成**一般危害** | ≈ L1-L2（公开-内部） |

### 2.3 分级判定矩阵（核心逻辑）

| 影响对象 | 特别严重危害 | 严重危害 | 一般危害 |
|---------|:----------:|:------:|:------:|
| **国家安全** | 核心数据 | 核心数据 | 重要数据 |
| **经济运行** | 核心数据 | 重要数据 | **敏感一般数据** |
| **社会秩序** | 核心数据 | 重要数据 | **敏感一般数据** |
| **公共利益** | 核心数据 | 重要数据 | **敏感一般数据** |
| **组织权益、个人权益** | **敏感一般数据** | **敏感一般数据** | **常规一般数据** |

> 关键洞察：矩阵的横轴是"危害程度"（特别严重/严重/一般），纵轴是"影响对象"（国家安全→个人权益）。这是平台自动化分级判定引擎的核心逻辑。

### 2.4 关键原则

- **就高从严**：数据集级别以所含数据项的最高级别为准。× 如果一张表里有 1 个核心数据字段 + 50 个常规字段 → 整表按核心数据保护
- **30% 阈值**：核心数据和重要数据的条目数量、存储总量等变化超过 30%，必须重新报送重要数据目录
- **动态更新**：业务属性、重要程度、危害程度变化时应及时更新分级，不是"一次分级终身不变"

### 2.5 六步操作流程

```
① 数据资源梳理 → ② 数据分类 → ③ 数据分级
                              ↓
⑥ 动态更新管理 ← ⑤ 报送重要数据目录 ← ④ 形成分类分级清单
```

### 2.6 金融数据分级示例（来自附录A推断）

| 三级分类 | 参考最低级别 | 理由 |
|---------|:----------:|------|
| 股票数据（公开行情） | 常规一般数据 | 公开信息，无危害 |
| 股票数据（大宗交易未公开） | 敏感一般数据 | 泄露影响市场公平 |
| 债券数据（公开） | 常规一般数据 | 公开信息 |
| 基金数据（公募） | 常规一般数据 | 公开净值 |
| 基金数据（私募未公开持仓） | 敏感一般数据 | 泄露影响投资者利益 |
| 外汇数据（涉及人民币汇率中间价未公开） | 敏感一般数据 | 影响经济运行 |
| 期货期权（公开行情） | 常规一般数据 | 公开信息 |
| 个人用户生物特征 | 重要数据 | 不可更改的敏感个人信息 |
| 个人用户交易数据 | 敏感一般数据 | 影响个人隐私 |
| 风控模型参数 | 重要数据 | 泄露影响金融稳定 |
| 系统安全配置（密钥、证书） | 核心数据 | 泄露危害国家安全 |

## 三、平台现状 vs 指南要求——差距分析

### 3.1 分类体系对比

| 维度 | 当前平台 | 指南要求 | 差距 |
|------|---------|---------|------|
| **分类框架** | 8 类业务导向（客户/交易/产品/财务/运营/风险/合规/系统），基于 JR/T 0197-2020 | 3 大类 9 中类 67 小类 | 🔴 结构性不匹配 |
| **层级深度** | 2 层（根类别 + 子类别） | 3 层（一级→二级→三级） | 🟡 需增加一层 |
| **金融数据标签** | 无 | 股票/债券/基金/外汇等金融产品维度 | 🔴 完全缺失 |
| **分类依据** | 业务领域（客户域/交易域等） | 数据性质（业务/用户/企业） + 金融产品类型 | 🔴 维度不同 |

### 3.2 分级体系对比

| 维度 | 当前平台 | 指南要求 | 差距 |
|------|---------|---------|------|
| **等级数量** | 4 级（L1-L4） | 4 级 | ✅ 数量一致 |
| **命名规范** | 公开/内部/敏感/机密 | 核心数据/重要数据/敏感一般数据/常规一般数据 | 🟡 命名不同，但可映射 |
| **判定逻辑** | 关键词+正则→规则引擎 | 影响对象×危害程度矩阵 | 🔴 判定维度完全不同 |
| **就高从严** | 无 | 必须 | 🔴 缺失 |
| **参考最低级别** | 无 | 附录A为每类数据预设 | 🔴 缺失 |
| **30% 变化阈值** | 无 | 有（核心/重要数据） | 🔴 缺失 |

### 3.3 流程对比

| 步骤 | 当前平台 | 指南要求 | 差距 |
|------|---------|---------|------|
| 数据资源梳理 | ✅ 数据源扫描+字段导入 | ✅ 要求 | 🟢 基本覆盖 |
| 数据分类 | ✅ 规则引擎+AI+人工 | ✅ 要求 | 🟡 分类维度需对齐 |
| 数据分级 | ✅ sensitivity_level | ✅ 要求 | 🟡 判定逻辑需升级 |
| 分类分级清单 | ❌ 无标准格式导出 | ✅ 必须 | 🔴 缺失 |
| 报送重要数据目录 | ❌ 无 | ✅ 必须 | 🔴 缺失 |
| 动态更新管理 | ❌ 无自动触发 | ✅ 30%阈值自动触发 | 🔴 缺失 |
| 人工复核 | ✅ ReviewRecord 工作流 | ✅ 隐含要求 | 🟢 已有基础 |

## 四、整合方案

### 4.1 总体思路

**不推倒重来，而是做"合规适配层"。** 当前平台的分类分级能力作为通用的数据治理层保留，在此基础上叠加一套"金融信息服务合规分类"（FinanceDataCategory），与现有 ClassificationCategory 建立映射关系。金融客户打开"合规模式"后，所有分类分级操作自动对齐指南标准。

### 4.2 实施分三个阶段

```
Phase 1（1-2天）：数据模型适配
  └─ 新增 FinanceDataCategory 表（存67类标准分类）
  └─ Field 表增加 finance_category_id + finance_data_level 字段
  └─ 新建两级映射表（67类 ↔ 现有8类）

Phase 2（2-3天）：判定引擎升级
  └─ 重写 TieringRule → 影响对象×危害程度 矩阵引擎
  └─ 实现"就高从严"（整表最高级别继承）
  └─ 实现"参考最低级别"默认值
  └─ 实现 30% 变化阈值检测

Phase 3（1-2天）：合规输出
  └─ 分类分级清单导出（指南标准格式）
  └─ 重要数据目录生成+报送
  └─ 动态更新提醒 + 合规仪表盘
```

### 4.3 Phase 1 详细设计：数据模型

**新增表：`finance_data_categories`**

```python
class FinanceDataCategory(Base):
    """金融信息服务数据分类（67类三级分类）"""
    __tablename__ = "finance_data_categories"

    id = Column(Integer, primary_key=True)
    code = Column(String(50), unique=True, nullable=False)  # 如 FIN_MKT_STOCK
    name = Column(String(200), nullable=False)               # 如"股票数据"
    level = Column(Integer, nullable=False)                  # 1=一级, 2=二级, 3=三级
    parent_id = Column(Integer, ForeignKey("finance_data_categories.id"))
    
    # 金融产品维度
    data_type = Column(String(50), nullable=False)           # business/user/enterprise
    finance_product = Column(String(100))                    # stock/bond/fund/forex...
    
    # 分级参考
    ref_min_level = Column(String(20), nullable=False)       # core/important/sensitive/normal
    level_rationale = Column(Text)                           # 为什么是这个级别
    
    # 附录A 内容
    appendix_desc = Column(Text)                             # 数据描述
    appendix_example = Column(Text)                          # 数据示例
    
    # 映射到现有分类
    mapped_category_id = Column(Integer, ForeignKey("classification_categories.id"))
    
    # 标准引用
    standard_ref = Column(String(500), default="国信办通字〔2026〕2号")
    version = Column(String(20), default="2026-06")
    is_active = Column(Boolean, default=True)
```

**Field 表新增字段：**

```python
# 在 Field 模型中新增：
finance_category_id = Column(Integer, ForeignKey("finance_data_categories.id"), nullable=True)
finance_data_level = Column(String(20), nullable=True)  
# 取值: "core", "important", "sensitive", "normal"
```

**分级枚举统一：**

| 平台当前 | 指南标准 | 新枚举值 | 说明 |
|---------|---------|---------|------|
| L4（机密） | 核心数据 / 重要数据 | `core` / `important` | 需从 L4 中拆分出"特别严重"和"严重" |
| L3（敏感） | 敏感一般数据 | `sensitive` | 基本对应 |
| L2（内部） | 常规一般数据 | `normal` | 注意：指南的"常规"涵盖了平台 L1+L2 |
| L1（公开） | 常规一般数据 | `normal` | 同上 |

### 4.4 Phase 2 详细设计：判定引擎

**核心：从"关键词匹配"升级为"矩阵判定"**

```python
# 影响对象枚举
IMPACT_TARGETS = [
    "national_security",    # 国家安全
    "economy",              # 经济运行
    "social_order",         # 社会秩序
    "public_interest",      # 公共利益
    "org_rights",           # 组织权益
    "personal_rights",      # 个人权益
]

# 危害程度枚举
IMPACT_LEVELS = [
    "extremely_serious",    # 特别严重
    "serious",              # 严重
    "general",              # 一般
]

# 判定矩阵（来自指南 §5）
GRADING_MATRIX = {
    ("national_security", "extremely_serious"): "core",
    ("national_security", "serious"): "core",
    ("national_security", "general"): "important",
    ("economy", "extremely_serious"): "core",
    ("economy", "serious"): "important",
    ("economy", "general"): "sensitive",
    ("social_order", "extremely_serious"): "core",
    ("social_order", "serious"): "important",
    ("social_order", "general"): "sensitive",
    ("public_interest", "extremely_serious"): "core",
    ("public_interest", "serious"): "important",
    ("public_interest", "general"): "sensitive",
    ("org_rights", "extremely_serious"): "sensitive",
    ("org_rights", "serious"): "sensitive",
    ("org_rights", "general"): "normal",
    ("personal_rights", "extremely_serious"): "sensitive",
    ("personal_rights", "serious"): "sensitive",
    ("personal_rights", "general"): "normal",
}
```

**判定流程：**

```
1. 分类阶段
   字段信息 → 匹配67类标准分类
   ├─ 规则引擎：字段名/表名/描述关键词 → 匹配到具体三级分类
   └─ AI：LLM 理解字段语义 → 补充分类
   → 得到 finance_category_id（如"股票数据"）
   → 读取该分类的 ref_min_level（如"normal"）

2. 分级阶段（三步判定）
   Step 1: 自动获取参考最低级别（从附录A映射表读取）
   Step 2: 检测是否触发"升级条件"
     - 是否涉及大量个人数据（>10万条）？
     - 是否涉及未公开信息？
     - 是否是系统安全类数据？
     → 有则上调一级
   Step 3: 人工确认（ReviewRecord 工作流）
   
3. 就高从严
   T = 表级最高级别 = MAX(该表所有字段的 data_level)
   如果 T ≥ "important" → 整表标记为 T 级保护
```

## 七、结论

**《金融信息服务数据分类分级指南》对平台不是"合规负担"，而是"产品化加速器"。**

1. 指南首次将金融数据分类分级标准化、量化（67 类 + 判定矩阵 + 30% 阈值），这使"自动化合规"成为可能，而自动化正是我们 AI 驱动平台的核心优势。

2. 当前平台架构可以低成本适配：ClassificationCategory（自引用树）和 TieringRule（规则引擎）的抽象设计是正确的。需要做的是"新增一层金融合规视图"，而非推翻重来。

3. 从市场时机看：指南 2026 年 6 月刚发布，大部分金融机构还在学习消化阶段。**现在是做"开箱即用的金融数据合规工具"的最佳窗口。**

**建议立即启动 P0（1 天），本周完成 67 类标准分类种子的预置。后续按客户需求节奏推进 P1-P3。**
