## 二、技术踩坑清单

### 2.1 前后端 URL 路径问题

**问题**：前端 `/dam` 子路径部署导致大量 307 重定向、401 认证丢失、静态资源 404。

**根因**：
- FastAPI 路由末尾不加 `/` 时，访问 `/path` 会 307 重定向到 `/path/`
- Vite 代理的 307 重定向会绕过代理回到原始域名
- Nginx 的 `/dam` 到 `/dam/` 的 301 导致 basename 丢失
- localStorage 的 token 在 `/` 和 `/dam/` 之间不共享

**解决方案**：
- 所有 API 路由加双装饰器：`@router.get("/path/")` + `@router.get("/path")`
- 前端 axios 拦截器主动加尾部斜杠
- Vite 代理配置 `proxy_redirect` 修正 Location 头
- localStorage key 命名空间隔离：`dam_token` vs `chamber_token`

**教训**：如果一开始就确定要部署在子路径，设计阶段就应该处理 URL 规范，不要到部署时再修。

### 2.2 SQLite vs PostgreSQL 的 Schema 同步

**问题**：本地 SQLite 开发一切正常，部署到 PostgreSQL 时报 `column does not exist`。

**根因**：
- SQLite 的 `Base.metadata.create_all()` 会自动创建缺失的表，但不会给已有表加列
- PostgreSQL 对缺失列的报错是硬错误（SQLite 在宽松模式下静默失败）
- Alembic 迁移脚本写了但一直是空 body，产物不可用

**解决方案**：
- 手动写 `ALTER TABLE ADD COLUMN IF NOT EXISTS` SQL
- 在 `deploy.sh` 中加入 schema 修复步骤（或统一用 `create_all` + 手动补列）

**教训**：
- 开发阶段就把 PostgreSQL 作为本地数据库，不要到部署时才切
- Alembic 要么用起来（写真正的迁移），要么删掉，空壳是最糟的

### 2.3 合规引擎的多层匹配设计

**问题**：初期设计只有关键词匹配，导致大量字段"未分类"。后续不断补关键词，变成打地鼠。

**根因**：低估了金融数据字段名称的多样性和中文同义词问题。

**解决方案**：5 层递进匹配——
1. 精确关键词（最高置信度）
2. 金融产品信号（stock/bond/fund/...）
3. 业务域映射（风控/财务/人力/...）
4. 兜底推断（日志类/身份类/交易类/风控类/金融产品类）
5. 遍历评分（子串匹配 + 分数阈值）

**教训**：分类系统一定要有兜底层。不能假设关键词会覆盖所有情况。

### 2.4 前端构建在 2C2G 服务器 OOM

**问题**：`npm run build` 在阿里云 2C2G 服务器上内存不足被 kill。

**根因**：Vite/Rolldown 构建单线程也可能吃 1GB+ 内存。

**解决方案**：
- `export NODE_OPTIONS="--max-old-space-size=512"` 限制 Node 堆
- `npm install --silent` 减少输出开销
- 添加 swap（2GB）作为安全垫
- deploy.sh 中先构建再部署，而非热更新

**教训**：低配服务器的前端构建要做内存限制，2GB swap 是底线配置。

### 2.5 动态置信度的设计陷阱

**问题**：初版置信度固定为 70%（命中关键词）或 60%（兜底），用户反馈"为什么都是 70%"。

**根因**：置信度应该反映匹配质量，而非简单的二分。

**解决方案**：`base 0.60 + cat_matched 0.10 + layer_bonus (0.20/0.15/0.10) + signal_bonus 0.05 + upgrade_bonus 0.05 + table_bonus 0.05`，上限 0.99。

**教训**：用户看到的任何数字，如果千篇一律，就会失去参考意义。置信度必须有区分度。

### 2.6 Windows 终端的编码问题

**问题**：Python 脚本输出中文到 Windows 终端时乱码，GBK 无法编码部分 Unicode 字符。

**解决方案**：用 Git Bash（`bash` 工具）而非 PowerShell 执行含中文输出的 Python。或者在 Python 中显式设置 `sys.stdout.reconfigure(encoding='utf-8')`。

**教训**：Windows 开发环境下，中文输出优先用 Git Bash。

## 三、验证有效的模式

### 3.1 标准文档 → 数据模型 → 引擎的转换链

**模式**：拿到一份法规/标准 → 先建数据模型（表）→ 再写引擎逻辑 → 最后做 UI。

本项目中：国信办通字〔2026〕2号 → FinanceDataCategory (67类) + FinanceGradingRule (18格矩阵) → ComplianceEngine (5层匹配) → StandardPage (3栏级联选择器)。

**可复用**：任何"标准/法规 → 软件功能"的需求都适用此模式。先建模，再写逻辑。

### 3.2 种子数据 = 开箱即用的关键

67 类标准分类不是让用户自己配置的，而是通过 `seed.py` 预置。用户打开平台就能看到完整的分类体系。

**可复用**：面向合规/标准类产品，预置种子数据是核心价值。竞品需要客户"自己配规则"——我们开箱即用。

### 3.3 P0→P1→P2 分阶段交付

不是一次性做完所有功能，而是：
- P0（1天）：表 + 种子数据 → 平台立刻能展示合规分类
- P1（1天）：引擎替换 → 自动化判定
- P2（1天）：输出 + 阈值 → 合规闭环

每阶段结束都有可演示的产物。

**可复用**：任何合规/标准化产品都可以用这个节奏。第一阶段的数据模型和种子数据本身就是可交付的 MVP。

### 3.4 AI 辅助 ≠ AI 主导

AI（通义千问）只做兜底分类和建议映射，最终判定由规则引擎（ComplianceEngine）完成。AI 的建议进入 ReviewRecord，需人工审批。

**可复用**：在合规场景下，AI 应该是"建议者"而非"决策者"。规则引擎做决策，AI 做补位。

## 四、下次会做得不一样的事

| 问题 | 现状 | 下次怎么做 |
|------|------|-----------|
| 数据库 | SQLite 开发 → PG 部署，切换时 schema 不一致 | 开发阶段就用 PG，或用 Docker Compose 统一环境 |
| 迁移 | Alembic 空壳 | 删掉 Alembic 或写真正的迁移脚本 |
| 前端子路径 | 部署时才处理 `/dam/` | 设计阶段就确定部署路径，前端在 dev 模式下也模拟子路径 |
| API 设计 | 路由斜杠问题反复修 | 统一规范：所有路由以 `/` 结尾，不写双装饰器 |
| 测试 | 手动测试 | 至少写核心引擎的单元测试（ComplianceEngine.classify_fields） |
| 监控 | 无 | 至少加 health check + 日志告警 |
