> 依据：六部门《金融信息服务数据分类分级指南》（国信办通字〔2026〕2号，2026-06-08）
> 版本：v1.0 | 2026-06-15

## 一、分类规则（三层体系）

### 1.1 分类总览

```
一级（3类）          二级（9类）              三级（67类）
─────────────────────────────────────────────────────
                    ├─ 金融市场数据 ───────── 13类（股票/债券/基金/外汇…）
                    ├─ 宏观经济数据 ─────────  4类（GDP/CPI/就业/贸易）
业务数据 ───────────┼─ 组织机构数据 ─────────  3类（金融机构/上市公司/发债主体）
                    ├─ 行业指标数据 ─────────  2类（财务指标/景气指数）
                    └─ 资讯报告数据 ─────────  2类（资讯/研报）

                    ├─ 个人用户数据 ─────────  3类（基本信息/交易/生物特征）
用户数据 ───────────┘
                    └─ 机构用户数据 ─────────  2类（基本信息/交易）

                    ├─ 经营管理数据 ─────────  6类（财务/结算/人力/市场/风控/其他）
企业数据 ───────────┘
                    └─ 系统运维数据 ─────────  4类（配置/日志/安全监测/安全事件）
```

### 1.2 分类判定流程

```
输入：一个数据字段的元信息（字段名、表名、业务域、描述、数据类型、样例值）

Step 1 — 精确匹配
  字段名是否包含以下关键词？
  ├─ "姓名/身份证/手机号/邮箱/地址" → 个人基本信息
  ├─ "指纹/人脸/声纹/虹膜/基因"     → 生物特征识别信息
  ├─ "密钥/证书/密码/私钥/Token"    → 安全监测数据
  └─ "股票/债券/基金/汇率/期货"     → 对应金融市场产品

Step 2 — 产品信号匹配
  字段文本（含描述）是否出现金融产品信号词？
  ├─ "股票/Stock/equity/股东/K线"  → 股票数据
  ├─ "债券/Bond/收益率/久期"        → 债券数据
  ├─ "基金/Fund/净值"               → 基金数据
  └─ "评级/违约/PD/LGD"             → 信用数据

Step 3 — 业务域映射
  字段所属业务域（business_domain）映射：
  ├─ 客户域     → 个人用户数据
  ├─ 交易域     → 个人交易数据 / 金融市场数据
  ├─ 风控域     → 风险控制与监督数据
  └─ 系统/安全域 → 系统运维数据

Step 4 — 遍历评分（兜底）
  遍历所有67类三级分类，按字段名子串+描述关键词+表名匹配加权评分
  └─ 得分 ≥ 0.2 → 采用；否则 → 标记为"待人工分类"
```

## 二、分级规则（四级矩阵）

### 2.1 四级定义

| 级别 | 名称 | 判定标准 | 典型数据 |
|------|------|---------|---------|
| **核心数据** | Core | 对**国家安全**造成**特别严重或严重**危害 | 密钥、证书、核心网络拓扑、国防军工数据 |
| **重要数据** | Important | 对**经济运行/社会秩序/公共利益**造成**严重**危害；或对国家安全造成**一般**危害 | 风控模型参数、反洗钱规则、生物特征、系统安全配置 |
| **敏感一般数据** | Sensitive | 对**经济运行/社会秩序/公共利益**造成**一般**危害；或对**组织/个人权益**造成**严重**危害 | 交易流水、持仓、个人身份信息、未公开市场数据 |
| **常规一般数据** | Normal | 对**组织/个人权益**造成**一般**危害 | 公开行情、产品说明、公告信息 |

### 2.2 分级判定矩阵

> 使用方法：先确定数据涉及的**影响对象**和可能的**危害程度**，然后查表。

| 影响对象 ↓ / 危害程度 → | 特别严重危害 | 严重危害 | 一般危害 |
|:------------------------|:----------:|:------:|:------:|
| **国家安全** | 🔴 核心 | 🔴 核心 | 🟠 重要 |
| **经济运行** | 🔴 核心 | 🟠 重要 | 🟡 敏感 |
| **社会秩序** | 🔴 核心 | 🟠 重要 | 🟡 敏感 |
| **公共利益** | 🔴 核心 | 🟠 重要 | 🟡 敏感 |
| **组织权益** | 🟡 敏感 | 🟡 敏感 | 🟢 常规 |
| **个人权益** | 🟡 敏感 | 🟡 敏感 | 🟢 常规 |

### 2.3 影响对象判定信号

| 影响对象 | 识别信号（字段名/描述中出现以下关键词时触发） |
|---------|------------------------------------------|
| **国家安全** | 密钥、证书、密码、加密、国防、军工、核心网络、关键基础设施 |
| **经济运行** | 利率、汇率、准备金、货币供应、基准、清算、人民币中间价 |
| **社会秩序** | 欺诈、洗钱、反恐、制裁、黑名单、恐怖融资 |
| **公共利益** | 上市公司、披露、重大事项、并购、退市、公众投资者 |
| **组织权益** | 模型、策略、参数、风控、评分、评级、额度 |
| **个人权益** | 身份证、手机号、姓名、地址、生物特征、指纹、人脸、账户、卡号、交易记录 |

### 2.4 危害程度判定信号

| 危害程度 | 识别信号 |
|---------|---------|
| **特别严重** | 密钥/证书/私钥/Token/核心系统 — 一旦泄露不可挽回 |
| **严重** | 生物特征/模型参数/未公开市场信息/大量个人数据（>10万条） |
| **一般** | 其他未触发升级的数据 |

### 2.5 参考最低级别（附录A 节选）

| 三级分类 | 参考最低级别 | 理由 |
|---------|:----------:|------|
| 股票数据（公开行情） | 常规 | 公开信息 |
| 股票数据（大宗交易未公开） | 敏感 | 泄露影响市场公平 |
| 债券数据（公开） | 常规 | 公开信息 |
| 基金数据（公募净值） | 常规 | 公开信息 |
| 基金数据（私募未公开持仓） | 敏感 | 泄露影响投资者利益 |
| 外汇数据（涉及人民币中间价） | 敏感 | 影响经济运行 |
| 个人基本信息 | 敏感 | 个人信息保护法 |
| 个人交易数据 | 敏感 | 金融隐私 |
| 生物特征识别信息 | 重要 | 不可更改 |
| 风险控制与监督数据 | 重要 | 影响金融稳定 |
| 安全监测数据 | 核心 | 危害国家安全 |
| 安全事件数据 | 重要 | 可被利用放大攻击 |
| 配置数据（网络拓扑） | 重要 | 影响系统安全 |

## 三、关键原则

### 3.1 就高从严

> **一个数据集（表）的安全级别 = 该表所有字段中的最高级别。**

示例：
```
表 dim_customer 包含：
  客户姓名      → sensitive
  身份证号      → sensitive
  生物特征模板  → important
  手机号        → sensitive

→ 整表 dim_customer 的级别 = important（取自生物特征模板）
```

这意味着：一条核心/重要级别的字段，会使整张表按最高级别保护。不能因为"只有一个字段是敏感的"而降低整表级别。

### 3.2 级别不可降

> 实际分级 ≥ 参考最低级别。只能上调，不能下调。

附录A 给出了每个三级分类的"参考最低级别"——这是底线。如果在判定过程中发现该数据涉及更高影响对象或更严重危害，必须上调。

示例：
```
外汇数据的参考最低级别是"敏感"。
但如果某外汇数据涉及人民币中间价未公开信息 → 影响对象="经济运行" → 可能上调至"重要"。
```

### 3.3 30% 变化阈值

> 核心数据和重要数据的**条目数量或存储总量**变化超过 30% 时，必须重新报送重要数据目录。

这是动态更新义务：
- 初始报送：首次分类分级完成后，将核心/重要数据报入重要数据目录
- 持续监控：系统每周自动检测核心+重要数据的条目数和存储量
- 触发条件：`|当前值 - 上次报送值| / 上次报送值 > 0.3`
- 触发后动作：生成合规变更提醒 → 人工复核 → 重新报送

### 3.4 动态更新

> 分类分级不是一次性工作。以下情况需要重新评估：

- 业务属性发生变化（如某字段从内部使用变为对外提供）
- 数据的重要程度发生变化
- 法律法规或标准更新
- 30% 阈值被触发

## 四、操作流程（六步法）

```
 ① 数据资源梳理
    盘点所有数据源 → 扫描所有表 → 列出所有字段
    ↓
 ② 数据分类
    每个字段按 §1.2 流程匹配到 67 类三级分类
    ↓
 ③ 数据分级
    每个字段按 §2.2 矩阵判定数据级别 + 就高从严
    ↓
 ④ 形成分类分级清单
    输出标准格式清单（字段名 → 分类 → 级别 → 判定理由）
    ↓
 ⑤ 报送重要数据目录
    将 core + important 级别数据提取为重要数据目录，上报监管
    ↓
 ⑥ 动态更新管理
    定期检测变化 → 30% 阈值预警 → 触发重新报送
```

## 五、附录：常见问题

**Q1：一个字段可能属于多个三级分类怎么办？**
A：选**最具体**的那个。比如"基金持仓数据"——既可以是"基金数据"也可以是"个人交易数据"。如果是从个人投资者视角（某个客户的持仓），归入"个人交易数据"；如果是基金产品视角（某只基金的持仓明细），归入"基金数据"。

**Q2：参考最低级别是"常规"的数据需要分级吗？**
A：需要。参考最低级别只是**底线**——在矩阵判定中可能因为涉及其他影响对象而升级。而且"就高从严"原则下，同表的其他字段可能拉高整表级别。

**Q3：已有的数据需要重新分类分级吗？**
A：是的。指南要求在指南发布后完成全量数据分类分级。但如果此前已按 JR/T 0197-2020 标准完成分类分级，平台支持建立新旧标准的映射关系，减少重复工作。

**Q4：什么数据不受本指南约束？**
A：国家秘密数据和军事数据。这两类数据有单独的保密法规管辖。
